# TOS Alpha-2.2.1 End-to-End Testing Roadmap

This roadmap defines the plan to graduate from the existing Playwright component tests (which mock the WebSocket connections) to a true End-to-End (E2E) testing suite that launches the production Rust Brain and dynamically interacts with real system state.

---

## Phase 1 — Framework Provisioning
*Establish the structural capability to spawn the Svelte UI and Rust backend in tandem within the Playwright runner.*

- [ ] **Dedicated Configuration:** Create `svelte_ui/playwright.e2e.config.ts`.
  - Override `testDir` to point to a new `svelte_ui/tests/e2e` directory.
- [ ] **Global Setup Script:** Create `svelte_ui/tests/e2e/globalSetup.ts`.
  - Compile the Brain via `cargo build`.
  - Spawn the brain daemon (`cargo run --bin tos-brain -- --headless`) bound to `localhost` and a guaranteed free testing port (or read dynamically via Brain IPC readiness probes).
  - Cache the PID for lifecycle management.
- [ ] **Global Teardown Script:** Create `svelte_ui/tests/e2e/globalTeardown.ts`.
  - Implement a graceful shutdown loop that sends SIGKILL/SIGTERM to the cached Brain PID, ensuring orphan background processes don't linger after failures.
- [ ] **Makefile Integration:** Add `make test-e2e` to the root `Makefile` to instantly bootstrap this integration environment and execute the suite.

---

## Phase 2 — Core Infrastructure Validation
*The baseline smoke tests confirming data correctly hydrates the Face from the Brain.*

- [ ] **Sanity Hydration Test (`tests/e2e/sanity.spec.ts`)**
  - Verify that the Svelte Face loads and the `connectionState` reliably resolves to 'connected'.
  - Verify no mock UI errors or console loops.
  - Assert that the Top Bezel system readout populates with the active actual Host OS identifier or Brain uptime stream.
- [ ] **Sector Mapping & Tree Sync**
  - Verify that `getDefaultState()` correctly yields to the first live `Sector` pushed over the WebSocket stream.
  - Assert that `globalOverview` tiles render dynamically matching real Rust sector manifests.

---

## Phase 3 — PTY Execution Matrix
*Deep integration paths verifying that shell inputs roundtrip from browser to system shell and back through WebSockets.*

- [ ] **Command Initiation (`tests/e2e/terminal.spec.ts`)**
  - Navigate to Command Hub (`Ctrl+2`).
  - Input a deterministic shell command (e.g., `echo "TOS_TEST_E2E_ALIVE"`).
  - Assert that the resulting `.terminalLine` DOM objects accurately contain the output block dispatched from the backend.
- [ ] **Process Registration**
  - Execute a long-running sleep command (e.g., `sleep 10`).
  - Assert that the UI renders an `activity-item` reflecting the live process state generated by `tos-priorityd`.
- [ ] **Tactical Context Menu Signals**
  - Trigger a right-click or long-press on the PTY task.
  - Fire the `[Signal] Force Kill` IPC action.
  - Assert the Rust backend tears down the Unix pipeline and the activity item subsequently is removed/flagged.

---

## Phase 4 — Edge Scenarios & Service Mesh Connect
*Testing system behaviors strictly contingent on the multi-daemon mesh environment.*

- [ ] **Trust Confirmation Blockers**
  - Stage `sudo su` via the terminal.
  - Ensure the PTY sequence is momentarily halted while the non-blocking amber trust chips are rendered dynamically overhead via the `TrustService`.
- [ ] **Split Detachment & Restore**
  - Split the viewport (`Ctrl+\`) and detach the container to a new sector.
  - Verify `tos-sessiond` dynamically writes the layout to the disk representation in `_live.tos-session`.
- [ ] **Marketplace AI Resolution**
  - Type a hallucinated error.
  - Ensure the `tos-heuristicd` backend daemon triggers semantic prompt chips dynamically in the prompt interlock overlay.
